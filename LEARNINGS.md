# LEARNINGS

Append-only log, one dated entry per session, added by running `/log-learning` at the end of each session (see `CLAUDE.md` Section 9 and `.claude/skills/log-learning/SKILL.md`).

Each entry has two sections: `Domain/Stack` (tech-stack/domain gotchas) and `Claude Code Usage` (what worked or didn't in the workflow itself).

## 2026-07-02 — Session 0

### Domain/Stack
- `tsx`/esbuild doesn't reliably emit the decorator metadata TypeORM needs to infer a column type from a bare `@Column()`. Every TypeORM `@Column()` must specify an explicit type string (`@Column('varchar')`, `@Column('uuid')`, `@Column('int')`, etc.) or the app crashes at startup with `ColumnTypeUndefinedError`. Now documented as a permanent rule in `CLAUDE.md`.
- Postgres `decimal`/`numeric` columns come back from `node-postgres`/TypeORM as strings, not numbers (precision-safety default) — a `ValueTransformer` is needed to get real JS numbers matching the API contract's shape.
- The `cors` npm package (v2.8.6), when given `origin: undefined` (e.g. an unset `CORS_ORIGIN` env var), does **not** fail open to a wildcard/permissive default — it fails *closed*, silently disabling CORS entirely so no `Access-Control-Allow-Origin` header is ever sent, blocking every cross-origin request including legitimate ones. Verified by reading the installed package source directly; four independent review-agent passes had all assumed the opposite (fail-open) direction.
- `tsx watch` on Windows has a flaky `EADDRINUSE` crash-loop under rapid successive file saves (multiple files edited within seconds of each other) — self-heals on the next file-save trigger. Confirmed unrelated to application code: added a shutdown handler suspected of causing it, then fully reverted it, and the same crash-loop still occurred in both states, then stopped once edits paused. Treat as tooling noise during a fast edit burst, not a code bug.
- Claude Code custom skills must live at `.claude/skills/<name>/SKILL.md` (a folder), not a flat `.claude/skills/<name>.md` file. The flat-file form silently fails to register as a slash command — no error, it just never appears in the skill list. `CLAUDE.md`'s own Section 2 project-structure diagram had documented the wrong (flat) layout; corrected there too.

### Claude Code Usage
- Multi-agent code-review consensus isn't proof — four separate finder angles independently reported the same wrong mechanism (CORS "wildcard" instead of "disabled") for the same bug. The verification pass that actually read the installed dependency's source code caught it. Trust primary sources over agent agreement when the claim is checkable.
- A plausible-sounding fix isn't necessarily the right one — reasoned that a graceful-shutdown handler explained the `tsx watch` `EADDRINUSE` issue, applied it, and the problem persisted; only reverting *and re-testing* (not just reverting) revealed the original theory was wrong. Worth re-verifying after reverting a suspected fix, not just assuming a plausible theory was correct.
- This session's user consistently corrected scope creep in real time — extra files/types/abstractions added without being asked (a `types/index.ts` before it was wanted, an env-validation helper addressing real bugs but seen as over-engineering) were reliably walked back. Staying strictly inside the currently-approved step/plan boundaries, and asking before adding anything beyond it — even small "obviously needed" additions — avoided repeated rework this session and is worth defaulting to from the start next time.

## 2026-07-03 — Session 1

### Domain/Stack
- TypeORM's `upsert()` typing breaks when an entity has a jsonb column typed `Record<string, unknown> | null` — `QueryDeepPartialEntity<T>`'s recursive mapping can't reconcile the `unknown` value type, forcing a `rows as QueryDeepPartialEntity<Place>[]` cast at the call site. Also: `QueryDeepPartialEntity` is exported from TypeORM's public `typeorm` entry point, not just the internal `typeorm/query-builder/QueryPartialEntity` path it's easy to reach for first — import from the public one.
- TypeScript's `moduleResolution: "node10"` deprecation warning's suggested silencing value (`ignoreDeprecations`) isn't stable across TypeScript versions — the project's local CLI TS wanted `"5.0"`, VSCode's bundled TS server wanted `"6.0"`. Chasing that value is a losing game. Migrating to `module`/`moduleResolution: "NodeNext"` + `isolatedModules: true` is the durable fix — and immediately surfaced the already-documented `import type` requirement for types used only in decorated property signatures (`Trip.ts`'s `TripPreferences` import had regressed to a plain import at some point).
- Vite has zero built-in Sass support — `.scss` files can exist and be imported with no error until real rules are added to them, at which point the build fails unless the `sass` package is installed. Easy to miss when placeholder `.scss` files were created empty in an earlier session and never actually exercised.
- Google Maps' `AdvancedMarker` (the modern marker API) requires a Map ID configured in Cloud Console; the legacy `Marker` component needs no such setup and is the right choice when a version doesn't need custom marker styling yet.
- Vite's `tsconfig.app.json` `"types"` array, once explicitly set (e.g. to `["vite/client"]`), suppresses automatic inclusion of *any* other installed `@types/*` package — even one pulled in transitively by a direct dependency (`@types/google.maps` via `@vis.gl/react-google-maps`). It has to be added to the array by name or the global `google` namespace won't resolve.
- React 19 + `verbatimModuleSyntax`: the ambient global `JSX` namespace no longer resolves for component return types — must `import type { JSX } from 'react'` explicitly.
- Google Places API (New) doesn't guarantee every `searchText` result has a populated `location`, even when explicitly requested via the field mask — the TS response type should mark it optional and calling code should filter, not assume the field mask is a hard guarantee.
- Express's `express.json()` middleware defaults to `strict: true`, which rejects a non-object/array top-level JSON body (e.g. a literal `null`) at the parser layer with an automatic 400 — a route handler doesn't need defensive code against `req.body` being `null`, because Express already refuses to hand it one.

### Claude Code Usage
- Multi-angle code review with a verification pass earns its cost even on a small diff: 8 finder angles surfaced roughly a dozen deduped candidates, and independent verification (actually running `tsc` against a hypothetical typo, spinning up a minimal Express instance to test a `null`-body claim) refuted several confidently-argued ones — a feared `req.body === null` crash, and two "make this more robust for future versions" suggestions that directly contradicted the project's own documented "don't build ahead of the current version" rule.
- Finder subagents working from a diff file (rather than fully reading the live file) can hallucinate line numbers that are obviously impossible (994, 221, 133 reported for files far shorter than that) — worth independently re-reading the actual current files to correct line numbers before reporting findings, rather than trusting agent-reported line numbers as-is.
- The user is comfortable redirecting architecture mid-session, not just at the initial planning stage — e.g. asking mid-step to extract page logic out of `App.tsx` into a dedicated `pages/TripPage.tsx` so `App.tsx` stays routing-only ahead of v7's real routing. Treating a checkpoint as genuinely open to structural feedback, not just a rubber-stamp, was the right posture and is worth continuing.
- After a code review, presenting findings with an explicit "what's actually worth fixing" recommendation — tiered by real impact vs. cosmetic — rather than just fixing everything or dumping the raw list, let the user make a fast, well-informed scope call instead of having to triage every finding themselves.
